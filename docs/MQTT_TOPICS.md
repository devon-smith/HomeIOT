# MQTT topic schema

Everything in the system flows through these topics. This is the contract between the orchestrator and adapters.

## Topic structure

```
home/{room}/{device}/state          adapter → bus, RETAINED, JSON
home/{room}/{device}/command        orchestrator → bus, NOT retained, JSON
home/_meta/adapter/{name}/health    adapter → bus, RETAINED, JSON
home/_events/{type}                 anyone → bus, NOT retained, JSON
```

### Naming rules

- `room` is a snake_case slug from `config/house.yaml` (e.g. `living_room`, `backyard`, `theater`).
- `device` is a capability slug, not a vendor name (e.g. `music`, `lights`, `climate`, `tv`, `water_feature`).
- One device family can map to multiple devices in a room. The Sonos in the living room is `home/living_room/music`; the lights are `home/living_room/lights`.
- House-wide things (no specific room) live under `home/_house/{device}` — used sparingly.

## State payloads

Required fields on every state message:

```json
{
  "ts": "2026-05-23T18:42:11.213Z",   // ISO 8601, UTC
  "source": "sonos-adapter",          // which adapter published
  "online": true,                     // device reachability
  "_cmd_id": "uuid-or-null",          // matches a command if this state is a response to one
  "pending": false,                   // true if a command is in flight
  "state": { ... }                    // capability-specific payload
}
```

`state` is capability-specific. Examples:

```json
// music
"state": {
  "playing": true,
  "track": "Live at Leeds",
  "artist": "The Who",
  "source": "spotify:playlist:abc",
  "volume": 30,
  "muted": false
}

// lights
"state": {
  "on": true,
  "brightness": 60,           // 0–100
  "scene": null,              // optional named scene currently active
  "color": null               // optional {h, s, v} or hex
}

// climate (hot tub, sauna, HVAC)
"state": {
  "mode": "heat",             // heat | cool | off | auto
  "target_f": 102,
  "current_f": 84,
  "heating": true
}

// tv
"state": {
  "on": true,
  "input": "hdmi3",
  "volume": 22,
  "muted": false
}

// water_feature
"state": {
  "on": true
}
```

State messages are **retained** so a restarting orchestrator gets the current world view from the broker the moment it subscribes.

## Command payloads

```json
{
  "id": "uuid",                       // generated by orchestrator, echoed back as _cmd_id
  "ts": "2026-05-23T18:42:11.213Z",
  "actor": "owner",                   // identity that initiated this
  "op": "play",                       // capability-specific verb
  "args": { ... }                     // capability-specific args
}
```

Examples:

```json
// home/living_room/music/command
{ "id": "...", "op": "play", "args": { "query": "jazz rock" } }
{ "id": "...", "op": "pause", "args": {} }
{ "id": "...", "op": "set_volume", "args": { "value": 25 } }

// home/living_room/lights/command
{ "id": "...", "op": "set", "args": { "on": true, "brightness": 60 } }
{ "id": "...", "op": "scene", "args": { "name": "evening" } }

// home/backyard/hot_tub/command
{ "id": "...", "op": "set_target", "args": { "target_f": 102 } }
```

Commands are **not retained**. They're fire-and-act-now messages.

### Correlation

The orchestrator generates `id` (UUID v4) when publishing a command. The adapter echoes that value back in `_cmd_id` on the next state publish if and only if that state change resulted from the command. State changes from external sources (Sonos app, C4 keypad, manual switch) publish state with `_cmd_id: null`.

This is how the orchestrator confirms user-initiated actions ("OK, playing X") versus reactively updating the world ("the user just paused from their phone").

## Adapter health

Every adapter publishes a heartbeat every 15s to `home/_meta/adapter/{name}/health`:

```json
{
  "ts": "2026-05-23T18:42:11.213Z",
  "name": "sonos",
  "version": "0.1.0",
  "uptime_s": 4827,
  "devices_online": 4,
  "devices_total": 5,
  "last_error": null
}
```

Retained, with the MQTT Last Will set so the broker publishes `{ "online": false }` if the adapter disconnects ungracefully.

## Cross-cutting events

`home/_events/{type}` is for things that don't belong to a single device.

```
home/_events/presence    { "actor": "owner", "home": true, "ts": "..." }
home/_events/schedule    { "job_id": "...", "fired_at": "...", "result": "ok" }
home/_events/scene       { "scene": "movie_night", "actor": "owner", "ts": "..." }
home/_events/approval    { "request_id": "...", "decision": "approved", "by": "owner" }
```

Not retained. The orchestrator subscribes broadly to `home/_events/#` for audit logging.

## Wildcard subscriptions in use

| Subscriber | Pattern | Purpose |
|---|---|---|
| Orchestrator | `home/+/+/state` | Maintain the world model |
| Orchestrator | `home/_meta/adapter/+/health` | Adapter liveness |
| Orchestrator | `home/_events/#` | Audit log |
| Each adapter | `home/{room}/{device}/command` for its devices | Receive commands |
| Debug tail | `home/#` | Everything (use `mosquitto_sub -t 'home/#' -v`) |

## QoS and retention

- State: QoS 1, retained
- Command: QoS 1, not retained
- Health: QoS 1, retained, with LWT
- Events: QoS 1, not retained

QoS 2 isn't worth the round-trips. QoS 0 loses messages on reconnect. QoS 1 (at-least-once) is the right default; idempotency is the adapter's responsibility.
